# GUI 智能体技术路线

## 一、总体思路

本项目面向缺乏 API 接口、难以通过命令行自动化的传统桌面软件或网页系统，构建一套基于 GUI 界面的智能操作代理。系统通过屏幕截图、OCR 文字识别、鼠标坐标、窗口状态和控件结构等信息感知当前界面，由大语言模型生成标准化操作指令，并在每一步操作后进行结果验证和错误诊断，实现"感知—规划—执行—验证—纠错"的闭环式自动操作。

该技术路线不依赖长期记忆库或重新训练模型，而是强调**任务执行过程中的在线错误分析与动态重规划**。当系统发现点击无效、输入失败、页面未跳转、弹窗遮挡或加载超时等问题时，会根据界面反馈自动分析失败原因，并调整后续操作策略，从而提高 GUI 自动化任务的稳定性和可控性。

## 二、技术路线总流程

```
用户任务输入
    ↓
任务解析（明确目标软件、操作目标和执行约束）
    ↓
隔离运行环境（虚拟机 / 远程桌面 / VNC / 沙箱桌面）
    ↓
界面感知（截图 + OCR + 鼠标坐标 + 窗口状态 + 控件树）
    ↓
目标识别与定位（识别按钮、输入框、菜单、弹窗、提示文字）
    ↓
动作规划（大语言模型生成结构化 JSON 操作指令）
    ↓
动作执行（点击 / 输入 / 滚动 / 快捷键 / 等待）
    ↓
结果验证（对比操作前后界面状态，判断是否达到预期）
    ↓
错误诊断（分析失败原因：坐标偏移、加载延迟、焦点错误、弹窗遮挡等）
    ↓
动态重规划（调整坐标、等待时间、识别方式或操作顺序）
    ↓
完成任务 / 人工接管
```

> 系统先看清界面，再判断要操作什么，执行后立刻检查结果；如果失败，就分析失败原因并重新规划，而不是盲目连续点击。

## 三、核心模块设计

### 1. 隔离运行环境模块

系统不直接控制用户主桌面，而是在虚拟机、远程桌面、VNC 或沙箱桌面中运行 GUI 自动化任务。

| 问题 | 解决方式 |
|------|---------|
| AI 操作影响用户正常使用 | 使用独立桌面会话 |
| GUI 操作必须可见 | 提供虚拟显示环境 |
| 操作失误风险较高 | 沙箱隔离、可随时中断 |

**实现方式：**
- Windows：RDP / 虚拟机 / pywinauto
- Linux：Xvfb / VNC / Docker GUI 环境
- 网页任务：Playwright / Selenium
- 通用坐标操作：pyautogui

### 2. 界面感知模块

| 感知对象 | 技术方式 | 作用 |
|---------|---------|------|
| 当前屏幕画面 | 截图 | 判断整体界面状态 |
| 鼠标当前位置 | 坐标采集 | 判断当前指针位置 |
| 页面文字 | OCR (Tesseract) | 识别按钮、标题、提示语 |
| 输入框和按钮 | UI Automation / 控件树 | 获取可操作元素 |
| 图标和固定区域 | OpenCV 模板匹配 | 定位视觉目标 |
| 页面变化 | 截图差分 | 判断操作是否生效 |

输出为结构化状态：

```json
{
  "window_title": "登录系统",
  "mouse_position": [640, 380],
  "detected_text": ["用户名", "密码", "登录", "忘记密码"],
  "elements": [
    {"type": "input", "label": "用户名", "bbox": [420, 260, 760, 300]},
    {"type": "input", "label": "密码", "bbox": [420, 320, 760, 360]},
    {"type": "button", "label": "登录", "bbox": [500, 390, 680, 430]}
  ]
}
```

### 3. 目标定位模块

采用多策略定位机制：
- **文字匹配**：OCR 精确/包含匹配
- **模糊匹配**：LCS 相似度算法，容许错别字
- **空间关系**：找到标签文字后定位其右侧/下方的操作元素
- **模板匹配**：OpenCV 图像模板匹配

### 4. 动作规划模块

LLM 输出标准化 JSON 动作指令：

```json
{
  "action_type": "click",
  "target": "登录按钮",
  "x": 590,
  "y": 410,
  "expected_result": "进入系统首页",
  "reason": "账号和密码已经填写完成，下一步应点击登录",
  "risk_level": "low"
}
```

### 5. 动作执行模块

执行优先级：
```
API / CLI > Playwright > UI Automation > OCR + 图像定位 > 坐标点击
```

### 6. 结果验证模块

| 验证方式 | 示例 |
|---------|------|
| OCR 文本变化 | 出现"登录成功""保存完成" |
| 页面跳转 | 窗口标题或 URL 变化 |
| 控件状态变化 | 按钮变灰、输入框填入内容 |
| 弹窗检测 | 出现错误提示或确认框 |
| 截图差分 | 页面局部区域发生明显变化 |
| 超时判断 | 等待后仍无变化，判定失败 |

### 7. 错误诊断与动态重规划模块

| 失败现象 | 可能原因 | 重规划策略 |
|---------|---------|-----------|
| 点击后无变化 | 坐标偏移 / 按钮未加载 | 重新识别按钮，调整坐标或等待 |
| 输入无效 | 输入框未获得焦点 | 先点击输入框，再重新输入 |
| 页面未跳转 | 网络延迟 / 登录失败 | 延长等待，读取错误提示 |
| 出现弹窗 | 弹窗遮挡操作路径 | 先处理弹窗 |
| OCR 未识别目标 | 字体小 / 背景复杂 | 改用控件树或图像模板匹配 |
| 页面状态不符 | 上一步操作失败 | 回退一步重新规划 |
| 连续失败 | 风险升高 | 停止任务并请求人工接管 |

## 四、完整技术路线表述

本项目采用"感知—规划—执行—验证—纠错"的闭环式 GUI 智能体技术路线。首先，系统在虚拟机、远程桌面或沙箱环境中构建独立 GUI 运行空间，避免智能体操作对用户主桌面造成干扰。随后，系统通过屏幕截图、鼠标坐标、窗口状态、OCR 文字识别、图像模板匹配和 UI Automation 控件树解析等方式获取当前界面信息，并将按钮、输入框、菜单、弹窗和提示文本等关键元素结构化表示。

在界面感知基础上，大语言模型根据用户任务和当前界面状态生成标准化 JSON 动作指令，包括点击、输入、滚动、快捷键和等待等操作。动作执行后，系统再次采集界面状态，通过 OCR 文本变化、页面跳转、控件状态变化、弹窗检测和截图差分等方式验证操作是否达到预期结果。

若操作失败，系统进入错误诊断与动态重规划流程，分析失败原因是否来自目标定位偏差、页面加载延迟、输入框未聚焦、弹窗遮挡、识别错误或页面状态不一致，并据此调整点击坐标、等待时间、识别方式或操作顺序。通过这一闭环机制，系统能够在不依赖长期记忆库、不重新训练模型的前提下，实现当前任务内的自适应纠错和稳定执行。
